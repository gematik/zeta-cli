# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this module is

`:connector` is a Kotlin library for talking to the gematik Konnektor (the Telematikinfrastruktur SOAP gateway). It bundles two distinct layers in one module:

- **`de.gematik.connector.api.*` — generated.** SOAP DTOs + `*Port` interfaces produced by `wsdl2openapi2kotlin` from `Konnektor-OPB6.json`. Every file under this package starts with `// Code generated by wsdl2openapi; DO NOT EDIT.` — don't hand-edit; the next regen will overwrite. Around 800 files.
- **`de.gematik.connector.*` (top of the package, not under `.api`) — hand-written.** The Kotlin port of koap-go's client surface: `Dotkon` config parser, `KonnektorClient`, `ServiceProxy`, `ServiceDiscovery` (service-directory loader), and the OkHttp TLS/auth helper at `de.gematik.connector.engine.okhttp`.

The boundary is the package: `.api.*` = generator output, everything else = client code. Do not put hand-written code under `.api.*`, do not put generated code outside it.

## Regenerating the API

The generator is an external tool (`wsdl2openapi`, kept in a sibling repo). Recipe lives in the root `justfile`:

```
just generate-connector
```

which runs:

```
java -jar ~/Development/gematik/wsdl2openapi/generator-kotlin/app/build/libs/wsdl2openapi2kotlin.jar \
    --file connector/Konnektor-opb6.json \
    --output connector/src/main/kotlin \
    --naming connector/naming-strategy.json
```

Inputs that *are* checked in and editable:

- `connector/Konnektor-OPB6.json` — the source spec.
- `connector/naming-strategy.json` — `basePackage` + regex `packageMappings` that flatten the upstream XML namespaces. Current rules: `de.gematik.ws.* → gematik.*`, `oasis.names.tc.x.y → oasis.xy`, `org.* → *`.
- `connector/build.gradle.kts` — `buildlogic.kotlin-library-conventions` + `kotlin("plugin.serialization")`, plus `ktor-client-core` (api), `kotlinx-coroutines-core`, `kotlinx-serialization-json`, `xmlutil-serialization`, and `ktor-client-okhttp` as `compileOnly` so the OkHttp bridge compiles without forcing OkHttp on every consumer.

The generator does not clean its output directory. After a regen with structural changes, wipe `src/main/kotlin/de/gematik/connector/api/` first and rerun — but **do not wipe other packages** under `de.gematik.connector/`, those are hand-written.

## Generated API layout (`de.gematik.connector.api.*`)

Subpackages reflect the upstream XML namespaces after `naming-strategy.json` mapping:

- `etsi022312`, `etsi01903132` — ETSI TS 102 231 / 101 903 v1.3.2 trust-list types.
- `gematik.conn.<service><version>` — Konnektor service ports and DTOs (`eventservice72`, `cardservice81…820`, `signatureservice74/75`, `certificateservice60/601`, `encryptionservice61/611`, `authsignatureservice74/741`, `cardterminal*`, `connectorcommon50`, `connectorcontext20`, …).
- `gematik.int.version.productinformation11`, `gematik.tel.error20` — gematik internal types.
- `oasis.dss10core`, `oasis.dssx10profiles.{signaturepolicy,verificationreport}` — OASIS DSS.
- `w3200009.xmldsig`, `w3200104.xmlenc` — W3C XML Signature / Encryption.
- `soap` — the one minimal hand-shaped surface generators emit: `SoapEnvelope` marker interface (`isFault(): Boolean`) and `SoapOperation(name, soapAction, bindingType)`. Every generated request/response has a `*Envelope` that implements `SoapEnvelope`.

Service port interfaces (`<Service>Port`) are plain interfaces, **not suspend**, with no Ktor/HTTP wiring. They exist for type completeness; the actual transport is `de.gematik.connector.ServiceProxy` (see below).

## Hand-written client (`de.gematik.connector.*`)

Layered on top of the generated `.api.*` types. Patterns are kept Kotlin/Native-friendly: stdlib `Base64`, no `java.io.File`, no JVM-only crypto types in pure data classes — JVM-only code (TLS / PKCS#12) is confined to `DotkonSecurity.kt` and `engine/okhttp/`.

- **`Dotkon.kt`** — `Dotkon` data class + `Credentials` sealed interface (`Basic`, `Pkcs12`). `parseDotkon(json, envLookup)` does `${VAR}` env-var expansion (callable lookup so it stays pure for Native), kotlinx-serialization decode with a `type` discriminator, then validates and throws `DotkonValidationException` collecting all field errors.
- **`ServiceDiscovery.kt`** — `ConnectorServices` and friends as xmlutil-serializable types matching `<base>/connector.sds`. `loadConnectorServices(httpClient, baseUrl)` is a suspend GET. `withRewrittenEndpoints(baseUrl)` returns a copy with internal hostnames replaced by the caller's reachable base. `defaultXml` is the shared `XML` instance used by both SDS and SOAP decoding.
- **`ServiceProxy.kt`** — `ServiceProxy.call(operation, request)` is the generic SOAP transport. Reified inline overload derives serializers from types: `val resp: GetCardsResponseEnvelope = proxy.call(Operations.GetCards, env)`. Does **not** check HTTP status (SOAP 1.1 transports faults via HTTP 500); callers either inspect `envelope.isFault()` directly or use `requireSuccess(operation)` which throws `SoapFaultException`.
- **`KonnektorClient.kt`** — high-level entry point. `KonnektorClient.connect(httpClient, dotkon)` loads the SDS and honours `rewriteServiceEndpoints`. Resolves `serviceProxy(name, version)` or `latestServiceProxy(name)` (highest semver). Convenience methods `getAllCards()`, `getCardsByType(...)`, `getResourceInformation()` mirror the koap-go surface.
- **`DotkonSecurity.kt`** — JVM-only, **engine-agnostic** translation of [Dotkon] into JSSE building blocks: `Dotkon.toTrustManager()`, `Dotkon.toKeyManagers()`, `Dotkon.toSslContext()`. Pure JDK, no Ktor — usable from any JSSE-backed HTTP client. The per-engine bridge below stays thin.
- **`DotkonAuth.kt`** — engine-agnostic Ktor extension `HttpClientConfig<*>.dotkonAuth(dotkon)`: pre-emptive HTTP basic auth via `defaultRequest`. PKCS#12 credentials are TLS client certs and are no-ops here.
- **`engine/okhttp/DotkonHttpClient.kt`** — thin Ktor/OkHttp bridge. `dotkonOkHttpClient(dotkon) { … }` builds an `HttpClient(OkHttp)`. `OkHttpConfig.dotkonTls(dotkon)` derives an `SSLSocketFactory` via `Dotkon.toSslContext()` and installs a `HostnameVerifier` for `insecureSkipVerify` / `expectedHost`. **SNI caveat:** OkHttp binds SNI to the request URL — `expectedHost` only adjusts hostname *verification*, not the SNI hint sent on the wire. `compileOnly` against OkHttp.

**Why OkHttp and not CIO:** Ktor 3.4.x CIO has its own custom TLS stack with hard-coded `CertificateType.RSA` / `DSS` only — no ECDSA. Gematik KSP / HBA / SMC-B card certs use brainpool ECC keys, which CIO silently filters out during cert selection, sending an empty `Certificate` message and failing mTLS at any nginx with `ssl_verify_client on`. OkHttp goes through JSSE, which sends what's installed. A previous CIO bridge was deleted for this reason. Adding another JSSE-backed engine (Apache, Java) is the same pattern: a thin file under `engine/<name>/` calling `dotkon.toSslContext()` and feeding the result into that engine's TLS config block.

### HTTP client ownership

The HTTP client is **always provided by the caller** — `KonnektorClient`, `ServiceProxy`, and `loadConnectorServices` all take an `HttpClient`, none of them constructs one and none closes it. `dotkonOkHttpClient` is offered as a convenience for the common .kon-driven case but is not required; any pre-configured Ktor `HttpClient` (any engine that goes through JSSE) works. This mirrors koap-go's split between `NewHTTPClient` (helper) and `NewClient` (user can supply their own `*http.Client`).

### XML namespacing

Every field in the generated DTOs carries an explicit `@XmlSerialName(name, namespace=…)` — **fields without an explicit namespace inherit the parent element's namespace** under xmlutil, which is stricter than Go's `encoding/xml` (which matches by local name only). When porting Go test fixtures verbatim, expect to add namespace prefixes to elements that Go was matching loosely.

## Build memory

The full module has ~800 generated source files. The Kotlin daemon needs more headroom than the default 512m to compile them — `gradle.properties` pins `org.gradle.jvmargs=-Xmx4g` and `kotlin.daemon.jvmargs=-Xmx4g`. If you see `OutOfMemoryError: GC overhead limit exceeded`, that's the cause; bump higher.

## Tests

`./gradlew :connector:test`. JUnit Jupiter via `buildlogic.kotlin-common-conventions`. Tests live in `src/test/kotlin/de/gematik/connector/` and use Ktor `MockEngine` for HTTP fixtures (no real Konnektor needed).

JVM bytecode disallows `.` in method names — so backtick-quoted test names (`fun `parses…dotkon`()`) must spell out "dotkon" rather than `.kon`.
